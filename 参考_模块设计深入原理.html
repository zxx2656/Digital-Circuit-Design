一、key_debounce — 为什么数字消抖是最优解

1.2 为什么不用 RC 硬件滤波

RC 滤波（电容+电阻）数字消抖（采样+计数器）
需要额外焊电容——元件清单里没有电容给你用纯 Verilog 代码，零硬件成本
RC 时间常数固定，调试时改不了——得换电阻/电容改采样次数只需改一个参数→重新下载，秒级迭代
电容有 ±20% 误差，每块板子消抖时间不一样FPGA 内部计数器绝对精确，无离散性
多路开关需要多个 RC 电路，占 PCB 面积Verilog 里多例化一个 instance，不额外占面积

1.3 采样频率为什么是 1KHz

这是一个折中选择：

约束分析

采样间隔 < 抖动最小宽度。 抖动脉冲最小 ≈ 0.5ms，所以采样间隔 ≤ 0.5ms → 频率 ≥ 2KHz。
采样间隔 > 消抖窗口/计数次数。 窗口=20ms，采样20次 → 间隔=1ms → 频率=1KHz。
不能太快。如果用 50MHz 直接采样——每个抖动毛刺都抓到，计数器不断被重置，永远消不干净。
不能太慢。用 100Hz→间隔10ms→可能连续两次采到抖动谷值→误判为稳定。

1KHz 刚好落在这四个约束的交集里。

1.4 消抖的深层问题：亚稳态

外部开关信号和 FPGA 内部时钟是异步的——开关随时可能跳变，跟时钟边沿无固定相位关系。如果跳变刚好发生在时钟上升沿附近：

亚稳态 (Metastability)

寄存器的建立时间 (setup time) 和保持时间 (hold time) 各约 0.1ns。时钟上升沿前后的 0.2ns 窗口内，输入不能变。
如果开关信号在这个窗口内跳变：寄存器输出既不是 0 也不是 1，而是中间电压——在接下来几百皮秒内悬在中间、随机振荡——最终随机落到 0 或 1。
两级同步器 (Two-stage synchronizer)：第一个寄存器可能进入亚稳态 → 第二个寄存器等一个时钟周期再采 → 这时第一个已经稳定了。

// 两级同步 + 消抖，完整链路 reg sync1, sync2; always @(posedge clk_1KHz) begin sync1 <= sw_raw; // 第一级：可能亚稳态 sync2 <= sync1; // 第二级：必然稳定 // sync2 才是安全的，送入消抖计数器 end

所以消抖模块的内部结构实际上是：两级同步器 → 连续采样计数器 → 输出更新。

1.5 消抖窗口的边缘情况

如果抖动超过 20ms（比如开关老化）

计数器会在每次抖动跳变时归零。如果抖动期 30ms，计数器会归零多次，最终在 30ms 后才确认稳定。
→ 用户会感到额外 10ms 延迟——毫无感觉。
→ 消抖不会失败，只是延迟更长。这是一个 自适应 的行为。

二、fsm_ctrl — 状态机设计的每一个决策

2.1 Moore 还是 Mealy

Moore 机Mealy 机
输出只取决于当前状态输出取决于当前状态 + 当前输入
输出与时钟同步变化——无毛刺输入变化时输出立刻变——可能产生毛刺
状态比 Mealy 多状态少但输出逻辑复杂
✓ 这个项目用 Moore不适合——LED 控制有毛刺会肉眼可见闪烁

Moore 机在这个项目里的体现：count_en / count_clr / led_run / led_pause 四个输出只由 state 寄存器决定，不看 sw_rst/sw_pause 的值。

2.2 为什么只有 3 个状态就够了

系统只有三种操作模式：

状态计数器数码管LED用户看到的
IDLE强归00:00显示00:00都灭秒表待命中
COUNT每秒+1实时变化绿灯亮正在计时
PAUSE冻结不动停在当前值红灯亮暂停了

有人会问：reset 是不是该单独一个状态？清零是一个动作而不是一种模式——进入 IDLE 时连带执行清零就完成了，不需要多一个状态。"预置"也不是一个独立状态——它只是 IDLE 状态下计数器初始值不为 00:00 而已。

2.3 状态编码

为什么用独热码 (One-Hot)

3 个状态，3 种编码方案：

编码IDLECOUNTPAUSE寄存器译码
二进制0001102bit ✓需要译码 ✗
独热码0010101003bit每bit=一个状态 ✓

用独热码：state[0]=IDLE, state[1]=COUNT, state[2]=PAUSE。
输出逻辑简化到极致：assign count_en = state[1]; // COUNT状态就是count_en
不需要译码器——状态寄存器的每一位直接就是一个输出。功耗更低、速度更快。

2.4 优先级编码的实现

// sw_rst 的优先级在 case 结构里体现 always @(posedge clk) begin case (state) IDLE: if (!sw_rst_clean) state <= COUNT; COUNT: if (sw_rst_clean) state <= IDLE; // ① 先判 else if (sw_pause_clean) state <= PAUSE; // ② 后判 PAUSE: if (sw_rst_clean) state <= IDLE; // ① 先判 else if (!sw_pause_clean) state <= COUNT; // ② 后判 endcase end

优先级由 if-else 的书写顺序决定，不是由什么"优先级编码器"电路决定。 综合工具按代码顺序生成逻辑——第一个 if 的条件被优先判断。

2.5 为什么输出是组合逻辑而非寄存器

两种方案对比

A：组合输出——state 一变 → next_state 一更新 → 同一拍输出跟着变。零延迟。
B：寄存器输出——state 变了 → 下一拍输出才更新。延迟一个时钟周期(1μs)。

1μs 延迟对秒表没任何影响，但组合输出更简单：
assign count_en = (state == COUNT);
assign count_clr = (state == IDLE);

一行代码，综合器优化成纯导线——不消耗 LUT。

三、counter_60 — 60进制BCD计数器的实现细节

3.1 为什么用 BCD 计数器而不是二进制 + 转换

两条路对比

方案A（二进制）： 16bit 二进制计数器 (0~3599) → 除60取模 → BCD转换 → 4位BCD码显示。
需要除法器（取模）或查 ROM 表，多消耗几十个 LUT。

方案B（BCD级联·本项目）： 4个独立的BCD计数器级联。每个位自己知道自己什么时候该归零。
零额外的转换电路，天然输出 BCD 码。

3.2 逐位分析

// 秒个位：核心逻辑 always @(posedge clk_1Hz or posedge count_clr) begin if (count_clr) sec_l <= 4'd0; // 异步清零优先 else if (count_en) begin if (sec_l == 4'd9) begin sec_l <= 4'd0; // 归零 carry_l <= 1'b1; // 产生进位 end else begin sec_l <= sec_l + 1; carry_l <= 1'b0; end end end

进位时序的关键

秒个位从9归零的同一拍，carry=1。
秒十位在下一拍clk_1Hz上升沿收到 carry=1→+1。
→ 进位是同步的，不是异步的。如果秒个位归零时异步触发秒十位+1→会出现毛刺和竞争。
同步进位：前一拍产生carry → 下一拍高位采样carry → 稳定可靠。

3.3 秒十位和分十位的特殊处理——满6归零

// 秒十位：不是满10，是满6 always @(posedge clk_1Hz or posedge count_clr) begin if (count_clr) sec_h <= 4'd0; else if (count_en && carry_l) begin // 上一级的进位到了才+1 if (sec_h == 4'd5) begin // 5→0，不是 9→0 sec_h <= 4'd0; carry_h <= 1'b1; end else begin sec_h <= sec_h + 1; carry_h <= 1'b0; end end end

3.4 59:59→00:00 的边界条件——两种实现方式

方式A：自然溢出方式B：显式检测（本项目）
等分十位从5归零→自然回到00:00 59:59时提前检测+强制归零
4级进位链，延迟4拍 1拍完成
进位链可能路径过长→时序违例 纯组合逻辑检测，无时序风险
// 显式检测：当所有位都到最大值时，下一个脉冲全部归零 wire is_59_59 = (min_h==4'd5) && (min_l==4'd9) && (sec_h==4'd5) && (sec_l==4'd9); always @(posedge clk_1Hz or posedge count_clr) begin if (count_clr) {min_h,min_l,sec_h,sec_l} <= 16'd0; else if (count_en) begin if (is_59_59) {min_h,min_l,sec_h,sec_l} <= 16'd0; // 全部归零 else // 正常级联计数... end end

3.5 预置值的实现

// 验收时改这两个参数即可 parameter INIT_MIN = 15; // 预置分钟 parameter INIT_SEC = 45; // 预置秒钟 // 复位时加载预置值 always @(posedge count_clr) begin if (count_clr) begin min_h <= INIT_MIN / 10; // 15 → 1 min_l <= INIT_MIN % 10; // 15 → 5 sec_h <= INIT_SEC / 10; // 45 → 4 sec_l <= INIT_SEC % 10; // 45 → 5 end end

上电后count_clr=1→counter_60内部归零条件触发→加载INIT值。用户拨开关1到"启动"→count_clr撤除→从INIT值开始计数。清零是回到00:00还是回到预设值，取决于count_clr行为。

四、seg_decoder — 扫描显示的每个细节

4.1 为什么是 1KHz 而不是 50Hz

人眼闪烁感知

临界闪烁频率 (CFF)：大约 50-60Hz。低于此频率→人眼感觉到闪烁。4位扫描→每位刷新率 = 扫描频率/4。
50Hz/4=12.5Hz→肉眼可见严重闪烁。
1KHz/4=250Hz→远超 CFF→人眼完全看不出闪烁。

为什么不用 10KHz？可以，但没必要。1KHz 已经远超 CFF，再高只会增加动态功耗（CMOS 电路功耗 ∝ 频率）。

4.2 残影（鬼影）问题——Break-Before-Make

// 扫描时序的状态机 always @(posedge clk_1KHz) begin case (scan_state) 2'd0: begin dig <= 4'b1111; // 先全关 scan_state <= 2'd1; end 2'd1: begin current_bcd <= bcd_inputs[scan_cnt]; // 换段选图案 scan_state <= 2'd2; end 2'd2: begin dig <= ~(1 << scan_cnt); // 再开目标位 scan_cnt <= scan_cnt + 1; // 准备下一位 scan_state <= 2'd0; end endcase end

3拍完成一次位切换：关(1拍)→换图案(1拍)→开(1拍)。 1KHz 时钟驱动，每位占 3 个时钟周期=3ms。4位×3ms=12ms 一轮≈83Hz 刷新率——刚好在人眼闪烁临界之上。

4.3 如果只有两拍（关→开在同一拍）

关和开之间没有专门的"换图案"拍→图案切换发生在开关切换的同一拍→旧图案短暂出现在新位上→残影。必须分开，不能省。

4.4 段选译码表的位序

SEG[7:0] 的位分配

数码管有 8 个段：a, b, c, d, e, f, g, dp。
SEG 是 8bit 总线——每个 bit 对应一个段。
具体哪个 bit 对应哪个段没有行业标准，由你自己在代码里定义。
定义完之后 PCB 接线必须照着这个定义来——Verilog 里 SEG[0]=a，PCB 上 SEG[0] 就必须接数码管 a 段。

// 一种位分配（推荐：按字母顺序） // SEG = {dp, g, f, e, d, c, b, a} 高位→低位 // a段=bit0, b段=bit1, ..., g段=bit6, dp=bit7 case (bcd) 4'd0: seg = 8'b1100_0000; // 0: abcdef亮 g灭 dp灭 4'd1: seg = 8'b1111_1001; // 1: bc亮 其余灭 4'd2: seg = 8'b1010_0100; // 2: abdeg亮 cf灭 ... endcase
